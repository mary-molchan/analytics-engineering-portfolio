# GenAI Prompt — Create Data Contract and Mapping

## Purpose

This prompt is used to transform a validated SQL Request Brief into a structured **Data Contract and Source-to-Target Mapping** for a Snowflake SQL stored procedure project.

The goal is to define source objects, target object expectations, field-level mapping, grain, joins, filters, transformation rules, data quality checks, and unresolved data dependencies before designing the stored procedure logic.

This prompt must not generate SQL code. It prepares the technical data specification required before procedure design.

---

## Input

Use this prompt with the validated output from Phase 1:

- [`../01_requirement_discovery/02_sql_request_brief_template.md`](../01_requirement_discovery/02_sql_request_brief_template.md)

Optional supporting inputs:

- source table documentation;
- target table structure;
- existing SQL query;
- data dictionary;
- sample records;
- column list;
- business rules;
- data quality rules;
- known Snowflake database / schema / table names.

## Prompt

```text
You are a Senior / Lead Data Engineer specialized in Snowflake SQL, data modeling, source-to-target mapping, and stored procedure design.

I will provide a validated SQL Request Brief for a Snowflake SQL stored procedure or complex SQL automation.

Your task is to create a Data Contract and Source-to-Target Mapping document.

The goal is to define the data requirements clearly before designing the stored procedure logic and before generating any Snowflake SQL code.

Important rules:
1. Do not write SQL code.
2. Do not design the full stored procedure logic yet.
3. Do not invent databases, schemas, tables, columns, data types, joins, filters, or business rules.
4. If information is missing, write "To be confirmed".
5. Use fully qualified Snowflake object naming when names are provided: DATABASE.SCHEMA.OBJECT.
6. Clearly separate confirmed data requirements from assumptions.
7. Identify the target grain and source grain.
8. Identify whether each target column is sourced directly, derived, defaulted, generated, or still undefined.
9. Identify join keys, join types, expected cardinality, and deduplication needs.
10. Identify filters, exclusion rules, and transformation rules explicitly.
11. Identify data quality checks required before insert, merge, update, or reload logic.
12. Identify risks related to missing columns, mismatched grain, duplicate records, null keys, ambiguous joins, late-arriving data, or unsafe target loads.

14. Do not assume that target table structure is valid unless confirmed.

## Human Review Checklist

After generating the Data Contract and Mapping document, the Data Engineer must verify that:

- no source object, target object, column, data type, join, or filter was invented;
- target grain is documented or marked as `To be confirmed`;
- every target column has a mapping status;
- join keys, join types, and cardinality are clear or marked as `To be confirmed`;
- duplicate risks and grain mismatch risks are visible;
- filter and transformation rules are explicitly documented;
- data quality checks are defined where needed;
- incremental, append, merge, or reload requirements are not assumed;
- Snowflake-specific considerations are identified where relevant;
- blocking open questions are clearly separated from non-blocking clarifications;
- the document is ready for [`../03_procedure_logic_design/`](../03_procedure_logic_design/).
```
